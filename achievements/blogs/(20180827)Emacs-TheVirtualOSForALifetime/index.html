Emacs - The Virtual OS for a Lifetime

An initial exploration of how I can use Emacs as a virtual OS while using Vim as a primary text editor.

2018/08/27

0. Introduction

At the previous article, I am introduced (and introduced) Vim (to myself too). When I do the research on the text editors which can satisfy my vast and strict requirements, two names came up, which are Vim and Emacs. I had tried both, and conclude that it is quite meaningless to compare two side by side. They have their strengths and weaknesses. It would be quite nice to have just one text editor for everything, but seems like we are not living in the world like that. There are always opponents, adversaries, rivals, antagonists, competitors, and challengers. Maybe this is the rule of the world! Without those, there will be less motivation. The Cold War ignited the competition between USSR and US, making the mankind send people to the moon, right (believe or not, wait, which part?).

Another conclusion I made is that Vim is the one that can make my hands and arms rest while typing, Emacs is not. There are a bunch of articles why Emacs is violating the ergonomic design concept, and I agree. I am writing this article and searching the Internet with Chrome browser with Vimium plug-in, and I haven't touched the mouse or Ctrl key. If I was editing this with Emacs, I would have given up already because of too much Ctrl and Meta (Alt) key combinations. (Please, please Emacs guys don't send a sniper...) The reason I started using Vim was the ergonomic design, the same reason I changed my keyboard from full keysets to 61 keys.

However, I think there are reasons why people use Emacs. According to my research, Vim users are complaining about Vimscript (why not Perl or Python?) and praising Emacs on its extensibilities. It seems like Emacs is more like an operating system because it has everything. No kidding, it has literally anything you can find in an OS. File system management, compilation and execution environments, modes, macros, buffers, and even Tetris (Do you see "Virtual Machine" at the menu bar?!)!

Therefore, my goal is to learn both of them. Ambitious? Well, you don't really know things until you learn it at a certain level, right? I am not saying I am going to learn them 100%, but I think I will have to learn them until I understand all the limitations and possibilities as a daily user. Then I can choose one, or just keep using both, or integrate one to another, or even make a new one which has all the advantages from the both.

In this article, I will open up my journey by sharing all the information I gathered so far. The format is almost same as the Vim introduction, but at the third section, I will emphasize the strengths of Emacs by going over all the points which make Emacs the best virtual OS, which also can be utilized as a supplementary part for Vim.

1. Emacs

1.1. What is Emacs

Emacs /ˈiːmæks/ is a family of text editors that are characterized by their extensibility. The manual for the most widely used variant, GNU Emacs, describes it as "the extensible, customizable, self-documenting, real-time display editor". - Wikipedia

It is really hard to call Emacs a text editor since it has so much more than that. So I am going to call it a virtual OS, but who really cares? It is what it is, and it would be enough to know what you can do with it. Let's see some advantages.

1.2. Advantages

The followings are advantages reported by numerous articles from the Internet:

  • Free Open Source Software
  • Multi-Platform Support (Linux, Mac, and Windows)
  • Fast and Light-Weighted Software with a Small Amount of Memory Footprint
  • Diverse Editing Functionalities (Internal Clipboard History, Mark-Rings, etc)
  • Modes (Programming, LaTeX, IRC, Email, Terminal, Server, Games, etc)
  • Extremely Extensible with its Own Scripting Language, ELisp
  • Seamless Integration with External Tools
  • Session Sharing and Remote Access
  • Built-in File Management System
  • Flexible and Customizable Features, Functions, and Macros
  • Active, Talented Development Communities

1.3. Disadvantages

The followings are disadvantages:

  • Lack of Some Desirable Features of ELisp
  • Multi-Threading Problems
  • Lack of Ergonomic Keybinding Design
  • Steep Learning Curve (Reportedly, Slightly Better than Vim)

Interestingly, I could see many people are actually using both for tasks where it can excel at. To use both applications at the same time, there are some solutions which I will explain in the last chapter.

3. Pressure Points

People might be wondering, what is the obsession about the ergonomic design and doing everything in a "text editor". A normal operating system and a set of programs will do everything that we need, then we don't need to use Vim and Emacs which are not pretty (sorry!) and user-unfriendly software. Then what is the reason that you, I, and we have to search the Internet "inside" of a text editor?

Ironically, the answer is the user interface. The "unified" user interface. A computer as a production tool for programmers, writers, and many text-based service providers and producers is nothing but a gigantic text editor. It is a very different story for graphic designers, songwriters, and sound designers. However, the former people do everything in text. Searching for information, writing text, modifying text, navigating text. File management, programming, project management, documentation, Internet browsing, email, chat, social network, news feed, etc. These tasks occupy 99%, and Emacs can do it without using external programs. For them, pixel numbers are meaningless, but text row and column numbers are essential.

The fact that you can do it everything inside of a text editor means you can use one unified user interface for all those activities. Also all those activities can be automated by the scripting language the editor provides. This aspect significantly reduces repetitive and mundane tasks, hence increases productivity. This is where Emacs shines.

Some might ask that you can do the same thing in Linux and Bash script. Partially true. Not because Linux and Bash are inferior to Emacs and ELisp, but the lack of standard and programmable interfaces. To write an extension in Emacs, the extension must give some interactive ways with the main Emacs part. Usually this is done by a programmable approach. For example, I can access the character located at row 12 and column 35 with ELisp or keybindings. However, can you do that easily for the Google Chrome in Linux? The answer is no. Usually the programs have different designs and it is not possible. There are some helper programs for the very purposes, but for 10 different applications you have to use 10 different helper programs. Do you see what I see?

Therefore, we (mostly me) need to maximize the strengths of the two programs by learning its best features. This is my personal preference / opinion so you don't have to agree. I think it is the best way for me. Other people who have strong arms/hands/wrists, they maybe prefer Emacs over Vim. I can't decide what suits for other people, and I think you shouldn't do that too. In the tech industry, there are too many egoistic wars to decide the best for something. A total waste of time, period.

Three main subjects will be investigated in the future, and here are the resources for that. This scheme might change in the future as I discover more features and possibilities. For Emacs, the focus is set to extensions and programmable interfaces (the scripting language ELisp). Finally, Vim mode in Emacs for my poor arms (and more productivity of course).
