1. Vim Text Editor

1.1. What is Vim Editor

Vim (/vÉªm/; a contraction of Vi IMproved) is a clone, with additions, of Bill Joy's vi text editor program for Unix. It was written by Bram Moolenaar based on the source for a port of the Stevie editor to the Amiga and first released publicly in 1991. Vim is designed for use both from a command-line interface and as a standalone application in a graphical user interface. - Wikipedia

It has a deep, vast history and will be a 1000-pages book if someone describes all the details about it. The most important thing is that it has survived and is still widely used by many people. In a nutshell, it is a modal text editor with programmable capabilities. All atomic operations like moving the cursor or entering characters are made in a modular fashion, so it is possible to combine them to make a new behavior quickly and easily.

In my opinion, this is a scripting language for text editing. The difference between Vi* and other scripting languages is Vi* uses keyboard input instead of strings of commands. People might say "Why the hack I should a scripting language to edit text?". This is a very valid point actually, so let's see some advantages.

1.2. Advantages

It is really hard to predict all the possibilities on what Vim can do, so I collected viable opinions from the Internet. Many of the opinions have something in common, and they are like the following:

  • Fast and Light-Weighted with Small Memory Footprint
  • Completely Free and Open Source
  • Stable and Good for Editing Large Files
  • Minimum Hands Movement during Editing
  • Powerful Built-in Commands (Saves Key Strokes)
  • Extreme Customization Possibilities
  • Cross-Platforms (Linux, Mac, and Windows)
  • Default Text Editor for Linux Machines
  • Extensive Programming Support for Most of Languages
  • Ready-Made Plug-ins for Expanding Functionalities
  • Multiple Copy and Paste Buffers (Registers in a Vim's Term)
  • Active Communities and Technical Support
  • Fast and Reliable Remote Access (Or for Pair-Programming)

All nice shiny things to have as a programmer, right? However, most of the convenience comes with a price. In fact, this is the biggest disadvantage of all, the price or cost you have to pay (sooner or later) for the functionalities.

1.3. Disadvantages

The followings are disadvantages or complaints well-known:

  • Anti-Intuitive Keybindings and Shortcuts
  • Extremely Stiff Learning Curve
  • Individual Preferences

Vim (or Vi*, I will just call it Vim including all of the forks like NeoVim) is hard to learn. This is the main reason I haven't used it so far. But now, the motivation for productivity comes first, so I started learning. Earlier I said it is a scripting language right? Other scripting languages are sort of... intuitive because they are logical in nature. For example, you know the functionality of the "if" statement because it's... if statement. However, Vim ignores the logical sense of usage. Starting with the movement cursors, h, j, k, l. These keys are lined up straight, very different layouts from the standard "arrow" cursors. Because it is designed for optimized movement.

However when you master it, you are editing the text at the maximum speed you can ever imagine. I just started using Vim (a week passed) and I already feel I am much more productive and don't feel the pain in my arms. This is a great motivation for me to keep me overcoming all the obstacles for the learning process. But still, the learning curve is unbelievable. See the graph below.

Even people who have been using Vim for decades say they only know like 10% about it. But I think their productivity already exceeds that of the others who don't use it. Some people say programming is not about typing but thinking and etc. That's correct if you only make simple programs without documentation. If you see some code of popular frameworks, often the code is 10 lines but documentation of the code is 300 lines. It is not close to the reality to say programming is not typing even though it is conceptually correct.

Therefore, it wouldn't really hurt anyone to learn Vim, or try to improve productivity in text editing in a similar manner. I performed a search to find the best learning materials that would make you learn without much pain (some pain is unavoidable though).

3. Migration Strategy

So this is the most valuable part of this article. I couldn't find a single subject on the Internet about how to start using Vim as a "pre-existing person" other than a baby who can use Vim without any prejudice. What I mean by that is most of us who want to use Vim are the people who are already so much attached to the existing technologies. It is simply not possible to change our primary text editor or IDE to Vim on a single day. I have read many people switched to Vim and couldn't use it because they had to give up too many things they were capable with the editors/IDEs they had been using before. Especially in a production environment, if you are a person who customized for your daily needs, your IDE is already too much customized and the moment you change it, you will lose it. You can't risk your job like that right?

What we need is a good transition/migration plan or strategy. In this way, we don't have to give up everything at a single moment, but we can divide and manage risks in a modular manner (like Vim!). In the worst case you find out you are not a "Vim" person, you can simply go back to the original. Or if you really like your current IDE, you can just stay in the middle (I will explain it later). Let's see what options we have.

3.1. Virtual Machines

If you are ready to use the hardcore command-line Vim right away, I recommend using a virtual machine to isolate the environment first. Usually we use Git for the development so moving data is not a problem. The code is always on the Cloud, then you are just editing it from the remote machine. Vim has a philosophy like that of *nix systems, so the seamless integration with the OS can be done when the OS is *nix systems. With a virtual machine, even Windows users can try out the OS and Vim at the same time. If you like Vim, chances are very high you would like Linux.

3.2. IDEs

Regardless of what OS you are using, you use IDEs for programming like me. I found it is the best for me to use Vim mode in those IDEs for now without sacrificing what I can do now with those IDEs. Also I was very surprised that almost all IDEs I am using support Vim mode. The followings are IDEs that support Vim mode:

  • Github Atom
  • JetBrains IDEA
  • Kate's KDevelop
  • Code-Blocks
  • RStudio
  • Microsoft Visual Studio
  • Microsoft Office Word
  • LibreOffice Writer
  • Apple XCode
  • Eclipse
  • Netbeans

Some IDEs like JetBrains' support the Vim mode from the manufacturer so you don't have to look for the 3rd party plug-in. Usually these have better performance. One thing you should remember is Vim mode plug-ins are not Vim itself. Sometimes it doesn't support all functionalities Vim has. However, it's undeniable that Vim mode is a good way to learn and prepare the complete migration to Vim.

4. (Bonus)Web Searching with Vimium

This is for the people who don't want to use a mouse for daily tasks. For me, using a mouse simply hurts my arms. Also it is very slow and cumbersome. Thankfully, there is a Chrome plug-in (extension) that enables to surf the web with a keyboard. It's not 100% perfect, but it significantly saves time. Try it!
