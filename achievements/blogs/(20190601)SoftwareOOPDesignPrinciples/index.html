Compiles major software design principles to provide a bird-eye view pervasive object-oriented design concepts in one place.

0. Introduction

It wouldn't be much of troubles to say the software world is being dominated by OOP. Of course, there are some other paradigms like imperative, declarative, procedural, functional, *-driven, or aspect-oriented; however, one can't deny that OOP would be one of the most important concepts. The problem is that it seems like there is no unified well-established consensus in the industry in terms of concepts and principles. There are lots of sources talking about it, but still, there is no single well-outlined document. Hence, I decided to explore and put all the concepts in one document with the simplest examples explaining the essence of each.

The second problem seen frequently in the dev world is that there are too many details to explain an elegant and simple concept. I, myself is a software person and I truly understand it's all about details, but learning materials shouldn't contain too much of it. Usually, students including myself (will be at the end of my life) want to understand the concept as soon as possible, not get bombarded too many details that hinder their quick and high-level understanding. I do recognize that there should be some detailed examples to explain a complex concept to manifest, but the core principles are not those.

The equal-level elements -- for example, if I want to use some fields, each field is an equal-level element. -- will be limited to three unless there are very good reasons to justify to use more. Two are usually too small to represent the entirety of a domain, and more than four usually are too many. There are going to be a brief description of the concept, a UML diagram to overview the structure of classes, and finally snippets of code for explanation.

I will try my best to include dominant concepts used for the OOP programming and organize these in a way we can formalize to use to build software or to understand the existing projects. However, I can be biased and the concepts can be viewed differently by perspectives and circumstances. Please don't hesitate to reach out to me if there is something incorrect or omitted.

1. 4 Major OOP Principles

1.1. Abstraction

1.1.1. Concept

Abstraction is not only a crucial principle of object-oriented programming but also the foundation of the software engineer or computer science. Wikipedia says:

the process of removing physical, spatial, or temporal details or attributes in the study of objects or systems in order to focus attention on details of higher importance, it is also very similar in nature to the process of generalization; the creation of abstract concept-objects which are created by mirroring common features or attributes from various non-abstract objects or systems of study — the result of the process of abstraction.

Usually, there is a piece of code showing a simplified model to represent a complex object when it's explained. However, I'd like to show a Java interface to explain how you can abstract out a common trait from multiple entities. See the following UML and code snippets.

1.1.2. UML

1.1.3. Code


public interface Livable {

    void eat();

    void sleep();

    void move();
}


public class Person implements Livable {
    @Override
    public void eat() {
        System.out.println("Person eats.");
    }

    @Override
    public void sleep() {
        System.out.println("Person sleeps.");
    }

    @Override
    public void move() {
        System.out.println("Person moves.");
    }

    public void talk() {
        System.out.println("Person talks.");
    }
}


public class Dog implements Livable {
    @Override
    public void eat() {
        System.out.println("Dog eats.");
    }

    @Override
    public void sleep() {
        System.out.println("Dog sleeps.");
    }

    @Override
    public void move() {
        System.out.println("Dog moves.");
    }

    public void barks() {
        System.out.println("Dog barks.");
    }
}

There are two concrete classes, Dog and Person. The common traits are extracted to form an interface. This is a process of abstraction, which enables to designate these classes (entities) as a generalized concept. Let's say we want to make a (programmable) house where a person or a dog can live. How do you express this in code? 


public class House {

    public static void main(String arg[]) {
        Person person = new Person();
        new House(person);

        Dog dog = new Dog();
        new House(dog);
    }

    private Livable livable;

    public House(Livable livable) {
        this.livable = livable;
        livable.sleep();
    }
}

Because of the power of abstraction, we could put anything in House when we think it's appropriate to relate entities to House. This principle is applied to many places in the OOP design principles, such as high cohesion or low coupling. House is not really coupled with any of concrete classes (Person, Dog) since we don't directly relate them to House. This means even if there are some changes in the concrete classes, it will be fine as fas as there is a promise (the interface) between them.

1.2. Encapsulation

1.2.1. Concept

Encapsulation helps to hide all the complexity of the inner mechanics of a class or an entity. Even though we abstract out an entity to make a model, which is a simplified representation of a real-world object, The users of the object shouldn't know all the details of the object. In many cases, there are some security problems when the users of the object can directly access to the inner states of the object.

Hence, it would be good to intentionally hide the inner complexity and expose a purified interface from and to the outer world, so the users of the object can interact with it in a way in which the creator or the manager of the object intended to. In this context, the developers of the object will encapsulate the object with a controlled interface.

1.2.2. UML

In this UML diagram, you can see each entity encapsulate itself with exposed interfaces, which are public methods. Users of these objects shouldn't and cannot know all the states in the objects. The only thing they can know is the exposed methods that enable the users to interact in a specific way defined by the creator of the objects.

1.2.3. Code

See how getters and setters are implemented in the following code. Since this pattern happens very frequently, some high-level languages support convenient ways to save typing -- e.g., getters and setters in Typescript. 


public abstract class Animal {

    protected String name;

    protected int speed;

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    public String getSpeed() {
        return speed;
    }

    public void setSpeed(String speed) {
        this.speed = speed;
    }

    abstract public void run();
}

1.3. Inheritance

1.3.1. Concept

Things in the universe -- or you might be able to expand this logic to the multiverse -- have hierarchies. Particular entities are more similar to related entities compared to entities far from its existence. For example, humans and dogs are animals, so two entities have many attributes in common; however, humans are less similar to insects compared to dogs.

This similarities and differences between entities can be systematically organized in code with inheritance. Inheritance enables to construct a level of abstraction existing among entities both vertically and horizontally. It also makes a computer (code) smart enough to blur the boundaries of concepts to allow computers to operate higher-level behaviors like humans.

1.4. Polymorphism

1.4.1. Concept

Whereas inheritance focused to express the similarities between entities, polymorphism is a way to represent the differences between entities. The behaviors specific to the concrete entities should be diverse depending on the context of the invocation of the behavior even though the behaviors have the same manner of invocation.

It is important to point out that the invocation occurs from the more abstract entities. For example, I know there is a variety of animals such as dogs, cats, humans, lions, etc. and I want them to run. Even though there are hundreds more, all I have to know is they are animal and they can run. As I instruct them to run only once, they will all run in a different way -- their own way. This type of command powers an instruction to handle explosively large cases without knowing all the details.

1.4.2. UML

Humans and dogs can run, but the behavior can quite be different. The dog entity and the human entity can show different characteristics when they are asked to show the same kind of behavior.

1.4.3. Code

The following code represents how to use polymorphic method calls by using a higher-level reference. 


public class Main {
    public static void main(String args[]) {
        Person person = new Person();
        person.setName("John");
        person.setSpeed("slow");

        Dog dog = new Dog();
        dog.setName("Toby");
        dog.setSpeed("fast");

        Animal animal;

        animal = person;
        animal.run(); // Mr. John runs slow

        animal = dog;
        animal.run(); // Spitz Toby runs fast
    }
}

2. GRASP (General Responsibility Assignment Software Patterns)

2.1. Controller

2.1.1. Concept

The controller handles all the system events and user input to connect them to the corresponding services. The controller separates the presentation logic from the pure domain area, which makes each component more reusable.

This controller pattern is well known as the MVC pattern, which stands for models, views, and controllers. There are some variations such as MVP (Model, View, and Presenter), PM (Presentation Model), and MVVM (Model, View, View-Model); however, these patterns aim to abstract similar parts of the application to increase cohesion and decrease coupling.

2.1.2. UML

The user interfaces get the input from the user and send the request to the controller. The controller gets the requests and parses to find which services it should redirect to. After getting the proper service for the view layer, the controller returns the result to the view.

2.1.3. Code


public class UIView {
    public static void main(String args[]) {
        System.out.println(Controller.handleOperation(1, 2, "add")); // 3
        System.out.println(Controller.handleOperation(5, 3, "subtract")); // 2
    }
}


public class Controller {
    public static Integer handleOperation(int a, int b, String operation) {
        if (operation.equals("add")) {
            return Addition.add(a, b);
        } else if (operation.equals("subtract")) {
            return Subtraction.subtract(a, b);
        }
        return null;
    }
}


public class Addition {
    public static int add(int a, int b) {
        return a + b;
    }
}


public class Subtraction {
    public static int subtract(int a, int b) {
        return a - b;
    }
}

2.2. Creator

2.2.1. Concept

Creation of objects is one of the most common activities in an object-oriented system. Which class is responsible for creating objects is a fundamental property of the relationship between objects of particular classes. In general, a class B should be responsible for creating instances of class A if one, or preferably more, of the following apply:

  • Instances of B contain or compositely aggregate instances of A.
  • Instances of B record instances of A.
  • Instances of B closely use instances of A.
  • Instances of B have the initializing information for instances of A and pass it on creation.

2.2.2. UML

The Person class is aggregated by the Company class, hence, the Company class should create the Person classes.

2.2.3. Code


public class Person {
    private String name;
    private String hobby;

    public Person(String name, String hobby) {
        this.name = name;
        this.hobby = hobby;
    }

    public void introduce() {
        System.out.println("My name is " + name + " and my hobby is " + hobby + ".");
    }
}


import java.util.ArrayList;
import java.util.HashMap;
import java.util.Map;

public class Company {

    private HashMap employeeList = new HashMap<>();

    public Company() {
        this.employeeList.put("Alice", "Swimming");
        this.employeeList.put("Carl", "Cooking");
        this.employeeList.put("James", "Walking");
    }

    public ArrayList enrollEmployees() {
        ArrayList people = new ArrayList<>();

        for (Map.Entry entry : this.employeeList.entrySet()) {
            people.add(new Person(entry.getKey(), entry.getValue()));
        }

        return people;
    }

    public static void main(String args[]) {
        Company company = new Company();
        for (Person person : company.enrollEmployees()) {
            person.introduce();
        }
    }
}

2.3. Indirection

2.3.1. Concept

The objective of this principle is to make objects less depending on each other by introducing a mediator object.

One example is the controller class between the view and the model class. Instead of having controller logic in the view class to call the corresponding services in the model, the controller introduces a layer between the model and the view. This is an indirection approach to decouple two layers and ensure the cohesive classes.

There are many use cases where you can introduce a layer between two functional units. Let's see one more example with code.

2.3.2. UML

The following example can show how the code gets simple and efficient by having another layer of functionality. Initially, the Sale class had the database access logic inside it, so after the creation of the Sale class, it can simply call the function of the Sale to save the record in the database. However, it violates the high cohesion and low coupling principles of the OOP design, so a new DatabaseAccessObject class was introduced.

The following diagram shows the refactored classes. The Sale class now can have only properties and related methods to the domain model. All the database system-specific logic is separated to the DAO (Database Access Object) so even if there are changes in the database, or you have to change the database, only DAO can change but the domain models don't have to change. This is the benefit of indirection.

2.3.3. Code


public class Sale {
    private int saleId;
    private String customerName;
    private String productName;
    private long amount;

    public int getSaleId() {
        return saleId;
    }

    public void setSaleId(int saleId) {
        this.saleId = saleId;
    }

    public String getCustomerName() {
        return customerName;
    }

    public void setCustomerName(String customerName) {
        this.customerName = customerName;
    }

    public String getProductName() {
        return productName;
    }

    public void setProductName(String productName) {
        this.productName = productName;
    }

    public long getAmount() {
        return amount;
    }

    public void setAmount(long amount) {
        this.amount = amount;
    }
}


public class DatabaseAccessObject {

    public DatabaseAccessObject() {
        System.out.println("Database initialized.");
    }

    public void saveSale(Sale sale) {
        System.out.println("The following sale saved: " + sale.toString());
    }

    public void closeDbConnection() {
        System.out.println("Database terminated.");
    }
}

2.4. Information Expert

2.4.1. Concept

The information expert principle says the responsibilities to be given should be able to be performed well with the information the target class has. This principle usually strengthens the high cohesion principle since the responsibilities and the associated information reside in the same class.

2.5. High Cohesion

2.5.1. Concept

The high cohesion principle evaluates the objects are appropriately focused, manageable, and understandable. All the elements in the class should be closely related and the responsibilities in the class ought to be focused.

Even in the lower-level programming such as assembly or C programming, there are modules and individual procedures composing the program. It can be said this supports the high cohesion principle. It would be a nightmare if you have to maintain a program consisted of one gigantic function.

2.6. Low Coupling

2.6.1. Concept

Coupling is a measure of how strongly one element is connected to, has knowledge of, or relies on other elements. The low coupling principle is to assign responsibilities to support the following requirements.

There should be lower dependencies between objects. If an object only can operate on the existence of another object, there is a dependency between those. Usually, depending on concrete objects create more dependencies than depending on abstract classes or interfaces.

There should be fewer impacts when there are changes in other objects. This leads to the objects which can be reusable because you can't reuse an object if you have to change it when other objects change.

2.7. Polymorphism

2.7.1. Concept

The polymorphism principle in the GRASP pattern designates the location of the responsibility of the polymorphic behavior. The polymorphic responsibility should be assigned to the very class where the polymorphic behavior will be used.

For example, there is an eat() function in the Animal class. This Animal class is inherited to the Dog class and the Person class, and these classes use the eat() function. The eat() in the Dog class will be to eat like a dog, therefore, its polymorphic behavior should be implemented in the Dog class. So should the Person class.

2.8. Protected Variations

2.8.1. Concept

The protected variation is well known as the open/closed principle. The fixed responsibilities can be in the class in a normal way; however, the responsibilities which are flexible and unstable should be extracted from the usual location of the class.

In other words, the fixed responsibilities should be closed to the changes and the flexible responsibilities should be opened to the changes. The architecture of the system should support this requirement. Let's see an example.

2.8.2. UML

The translator contains several steps of logic, but one of the components should change according to the responsibility. In this example, the language should change depending on the user's preference, but the other parts of the translator shouldn't change.

We can make an inner class have a variation of the responsibilities and requirements to isolate the part that frequently changes. Even if there are additional requirement changes, fixed parts of the class don't have to change.

2.8.3. Code


public class Translator {

    public void translate(LanguagePack languagePack) {
        preTranslate();
        processLanguage(languagePack);
        postTranslate();
    }

    public void preTranslate() {
        System.out.println("Preprocessing...");
    }

    public void processLanguage(LanguagePack languagePack) {
        System.out.println("Process language...");
        languagePack.applyLanguage();
    }

    public void postTranslate() {
        System.out.println("Postprocessing...");
    }

    static class LanguagePack {
        private String language;

        LanguagePack(String language) {
            this.language = language;
        }

        public void applyLanguage() {
            System.out.println("Apply language with " + language);
        }
    }

    public static void main(String args[]) {
        new Translator().translate(new LanguagePack("English"));
        new Translator().translate(new LanguagePack("Korean"));
        new Translator().translate(new LanguagePack("French"));
    }

}

2.9. Pure Fabrication

2.9.1. Concept

Sometimes following the information expert principle violates the high cohesion principle and the low coupling principle. In this case, you can make a new class to put the responsibilities. This is the pure fabrication principle.

The purely fabricated class is a class that does not represent a concept in the problem domain, specially made up to achieve low coupling, high cohesion, and the reuse potential. The database access object can be an example since this logic should be inside of the domain model according to the information expert principle, but it decreases cohesion and increases coupling. Therefore, it would be better to make a separate class.

3. SOLID Principles

3.1. SRP – Single Responsibility Principle

3.1.1. Concept

The single responsibility principle states that every module, class, or function should have responsibility for a single part of the functionality and that responsibility should be entirely encapsulated by the class.

This is a simple principle; however, the more important question would be on how we define the scope of a single responsibility. Imagine we are making a car. It's possible to have a single Car class and all the processes in a single class. However, as you go deep into the details, you will realize the processes are much more complex. Then we could break down the processes, so it can have classes that make a wheel, make a handle, make a tire, make a frame, and more. What level of abstraction should we use?

There is no definite answer. But the rule of thumb would be it should be enough to be able to follow other object-oriented principles. If the level of the abstraction is too high, probably it will violate other principles such as high cohesion and low coupling. The code should be radically refactored as the complexity is discovered and it needs to change the level of abstraction.

3.2. OCP – Open/Closed Principle

3.2.1. Concept

This principle is the same as the protected variation principle in the GRASP pattern, so the explanation will be skipped.

3.3. LSP – Liskov Substitution Principle

3.3.1. Concept

The Liskov substitution principle is an ability to replace any instance of a parent class with an instance of one of its child classes without negative side effects.

The Liskov principle is closely linked to the open-closed principle which is open to the extension and closed to the modification since by moving down the class hierarchy, it is open to the extension but not the other way around.

All the behavior of the child classes should be extending the behavior of the parent classes, not replacing it. A conceptual example would be birds. There is an abstract class Bird and it has a fly method. A Duck class can extend it and override the fly method. A Chicken class can extend the Bird method but can't use the fly method, so it should prevent users to use the fly method by using exceptions or other types of means. This violates the Liskov substitution principle. Let's see another example with code.

3.3.2. UML

Squares have more restrictive requirements compared to rectangles, so it might make sense the Square class extends the Rectangle class. However, as you can see in the code, the Square class can't substitute the Rectangle class because you can't set the height and the width separately. This violates the Liskov substitution principle.

The right approach would be making a Shape class, then the Rectangle class and the Square class can extend the Shape class. In this way, both the Rectangle class and the Square class can substitute the Shape class, hence, doesn't violate the Liskov substitution principle.

3.3.3. Code


public class Rectangle {
    int height;
    int width;

    public int getHeight() {
        return height;
    }

    public void setHeight(int height) {
        this.height = height;
    }

    public int getWidth() {
        return width;
    }

    public void setWidth(int width) {
        this.width = width;
    }
}


public class Square extends Rectangle {

    @Override
    public void setHeight(int height) {
        super.setHeight(height);
        super.setWidth(height);
    }

    @Override
    public void setWidth(int width) {
        super.setWidth(width);
        super.setWidth(width);
    }
}

3.4. ISP – Interface Segregation Principle

3.4.1. Concept

The interface segregation principle states that no client should be forced to depend on methods it does not use. It's all about not making a gigantic interface that contains a large number of methods. The following example displays an incorrect usage of an interface.

3.4.2. UML

It's tempting to contain all possible method signatures in one interface so you can just manage all classes from an interface; however, this creates unnecessary implementations which makes the code inefficient as the classes using the interface must implement all methods regardless of the actual usage.

Conceptually, an animal can swim or fly, but if the interface has both of them, the classes using this interface should implement both even if they don't use all of them. It violates the interface segregation principle.

3.4.3. Code


public interface IAnimal {
    void swim() throws Exception;
    void fly() throws Exception;
}


public class Dolphin implements IAnimal {
    @Override
    public void swim() throws Exception {
        System.out.println("Swimming...");
    }

    @Override
    public void fly() throws Exception {
        throw new Exception("I don't fly.");
    }
}


public class Eagle implements IAnimal {
    @Override
    public void swim() throws Exception {
        throw new Exception("I don't swim.");
    }

    @Override
    public void fly() throws Exception {
        System.out.println("Flying...");
    }
}

3.5. DIP – Dependency Inversion Principle

3.5.1. Concept

The dependency inversion principle states that the high-level objects should not depend on the low-level implementations. Both should depend upon abstractions. By using the dependency inversion principle, the architecture will be less coupled.

3.5.2. UML

Web applications are good examples. Let's assume a controller depends upon a repository directly. It increases the coupling between the controller layer and the repository layer because it is not easy for the controller layer to change the repositories.

There can be an abstraction layer between two as an interface called IRepository. The controller depends on this interface and so does the repository. It doesn't violate the dependency inversion principle anymore.

3.5.3. Code


public interface IRepository {
    void create();
    void read();
    void update();
    void delete();
}


public class Controller {
    private IRepository iRepository;

    Controller(IRepository iRepository) {
        this.iRepository = iRepository;
    }

    public void create() {
        iRepository.create();
    }

    public void read() {
        iRepository.read();
    }

    public void update() {
        iRepository.update();
    }

    public void delete() {
        iRepository.delete();
    }
}


public class Repository implements IRepository {
    @Override
    public void create() {
        System.out.println("Created!");
    }

    @Override
    public void read() {
        System.out.println("Read!");
    }

    @Override
    public void update() {
        System.out.println("Updated!");
    }

    @Override
    public void delete() {
        System.out.println("Deleted!");
    }
}


public class Main {
    public static void main(String args[]) {
        IRepository iRepository = new Repository();

        Controller controller = new Controller(iRepository);

        controller.create();
        controller.read();
        controller.update();
        controller.delete();
    }
}

4. Uncategorized Principles

4.1. Common Closure Principle

4.1.1. Concept

The common closure principle states that the classes in a package should be close together against the same kinds of changes. A change that affects a package can affect all the classes in the same package but no other packages.

This is similar to the single responsibility principle, but the scope is not a single class but a package level. If there are classes in other packages being affected, maybe those classes should be included in this class. If not all classes in the package are affected by a single change, maybe those classes should be separated into another package.

4.2. Delegation Principle

4.2.1. Concept

The delegation principle states the classes should delegate responsibilities that are not specialized for the very class. In the real world, people depend upon other people in various ways.

A hamburger is made by some components like patties, bread, and vegetables. The person who makes a burger delegates those responsibilities to others. Farmers can make them but they also depend upon others for getting things needed to produce them.

These chains of delegation represent the entire supply processes with all involved entities or subject in an organized way. The code becomes manageable and extensible.

4.2.2. UML

See the following example to see how the carmaker can delegate the responsibilities to others with delegation:

4.2.3. Code


public class CarMaker {
    public static void main(String args[]) {
        new CarMaker().makeCar();
    }

    public void makeCar() {
        new FrameMaker().makeFrame();
        new EngineMaker().makeEngine();
        new WheelMaker().makeWheel();
    }
}


public class FrameMaker {
    public void makeFrame() {
        System.out.println("Frame is being made...");
    }
}


public class EngineMaker {
    public void makeEngine() {
        System.out.println("Engine is being made...");
    }
}


public class WheelMaker {
    public void makeWheel() {
        System.out.println("Wheel is being made...");
    }
}

4.3. Package Principle

4.3.1. Concept

The package principles are a way of organizing classes in larger systems to make them more organized and manageable. They aid in understanding which classes should go into which packages (package cohesion) and how these packages should relate with one another (package coupling). Let's see them in detail.

Principles of Package Cohesion

  • Reuse-release Equivalence Principle (REP): Packages must be created with reusable classes.
  • Common-Reuse Principle (CRP): Classes that tend to be reused together belong in the same package together.
  • Common-Closure Principle (CCP): The package should not have more than one reason to change.

Principle of Package Coupling

  • Acyclic Dependencies Principle (ADP): There can be no cycles in the dependency structure.
  • Stable-Dependencies Principle (SDP): Any packages one wants to be volatile should not be depended upon by a package that is difficult to change.
  • Stable-Abstractions Principle (SAP): A stable package should also be abstract so that its stability does not prevent it from being extended.

Package principles also include Software Package Metrics, which help to quantify the dependency structure, giving different and/or more precise insights into the overall structure of classes and packages.

4.4. KISS (Keep It Simple Stupid) Principle

4.4.1. Concept

This might sound very simple, but actually, it's not that easy. The real-world problems are complex, so it needs to be broken down to the simplest, smallest, and understandable pieces before even start designing the architecture. If this stage is omitted, the solution will most likely be complex and violating other OO principles too. There are practical guidelines to follow this principle:

  • Break down the tasks into sub-tasks taking no more than eight hours.
  • Break down the problems into sub-problems solvable with a few classes.
  • Each method should be no more than 20-30 lines of code and solve just one problem.
  • Each class should be following the same principle as the methods.
  • Solve the problem then start coding, don't solve the problem while coding.
  • Don't be afraid of discarding the code you wrote.

4.5. YAGNI (You Aren’t Gonna Need It) Principle

4.5.1. Concept

The YAGNI principle states that a programmer should not add functionality until deemed necessary. You should always implement functionalities when you actually need them, never when you just foresee that you need them.

This principle needs to be performed in the context of Agile development. In the Agile practice, the team should make the simplest thing that could possibly work in that stage. As more requirements are added, the code should be refactored to accommodate newer requirements.

4.6. DRY (Don’t Repeat Yourself) Principle

4.6.1. Concept

The simplest rule of all. Don't repeat the code, if there are repetitions, it needs to be resolved using other OO principles. By following the guidelines explained in the KISS section, one can achieve the DRY principle as well.

4.7. Inversion of Control (IoC) & Dependency Injection (DI)

4.7.1. Concept

The inversion of control principle is a technique to decrease coupling and increase reusability by inverting the flow of control in the program.

Usually, the object that uses another object makes another object in the object; however, in this way, the original object depends on another object and it gets heavy because it has to initialize the dependencies. If the flow can be inverted and the dependency can be injected to this object from a management system, we can decrease coupling and reuse the injected objects repeatedly.

4.7.2. UML

Originally, the database access object is made to save the Sale class's record. However, the database access object shouldn't have information about the Sale class since the DAO should have only information on the database system. Instead, the Sale class can have a reference of the DAO and the DAO object can be injected when needed. In this way, the DAO can be reused by other objects such as the Customer class or the Product class.

Furthermore, it is possible to make a Save class to take a record class and a DAO class as references to perform a save operation. By doing this, the database transaction itself can be decoupled from the business model so domain models don't have to know about the database logic anymore.

4.8. Composition Over Inheritance Principle

4.8.1. Concept

This is about how to define the relationships between entities in your program. There are many approaches, but the simplest one is:

  • Think of containment as a has a relationship. A car "has an" engine, a person "has a" name, etc.
  • Think of inheritance as an is a relationship. A car "is a" vehicle, a person "is a" mammal, etc.

Another way is to use the Liskov Substitution Principle (LSP) to determine which to use between inheritance and composition.

Here is the pros and cons of inheritance:

  • Code re-usability: Whatever code defined in a superclass can be reused by any of its subclasses.
  • Polymorphism: The superclass type variable can hold a reference to an object of superclass type or any of its subclass type.
  • Dynamic Binding: At run time JVM will decide which method to invoke based on class type of the object.
  • Ease of addition of new subclass: Suppose need to add new subclass then it can be added just by inheriting the superclass in the new class. It will affect neither superclass nor its existing subclasses.
  • Extra Baggage: Suppose you want to use only one method of a class but not others, then we have to inherit the class which comes with all other unwanted methods and data members as well.
  • Tightly Coupled: Superclass and subclass are tightly coupled. Changes made to a superclass can break the subclass.
  • Weak Encapsulation: The members are closely related to its superclass, so it is also opened to modification.

Here is the pros and cons of composition:

  • Loosely coupled: Changes in the class doesn't affect any members of the class.
  • Code re-usability: Code re-usability can be achieved the same as Inheritance but need to explicitly call the code which needs to be reused.
  • Strong Encapsulation: All members are closed to change since they don't have relationships to other classes.
  • More Difficult to add a new class: It is easier to add a new class in composition compared to Inheritance as the need to write code explicitly in a new class to reuse the existing ode.
  • Performance Cost: The explicit method call forwarding/delegation approach has a performance cost compared to inheritance single invocation.

0. Conclusion

It is all about the perspective on how we see the world, how we perceive the entities, how we recognize relationships between those entities. The object-oriented approach is not the only philosophy we should follow but seems like it is pervasive because it can solve the problems of the 21 century.
