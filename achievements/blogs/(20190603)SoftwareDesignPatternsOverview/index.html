Software Design Patterns Overview

Explore the variety of object-oriented design patterns used in the enterprise applications.

2019/06/03

0. Introduction

Design patterns are general repeatable solutions for commonly occurring problems in software projects. Since the engineers started using object-oriented principles, they have been accumulating solutions which can improve efficiency, scalability, and readability. These solutions were developed from the trials, errors, and mistakes of the past, so the community can use them to stand on the shoulder of a giant -- well, the software development community.

Design patterns have two major benefits. First, they provide a way to solve issues related to software development using a proven solution. Second, design patterns can be an effective communication mean between architects, business analysts, system administrators, and software engineers.

Usually, patterns are developed in the refactoring process because you can notice inefficient parts after finish writing code from the requirements; however, if you are knowledgeable in patterns, you can start applying patterns from the beginning. Can you imagine how efficient it will be if you can skip several refactoring processes?

Unfortunately, it requires experiences, knowledge, and efforts to be able to start translating the requirements into the known design patterns from the beginning. This trait can be seen only from architects who spent decades in the industry, but I think this can be learned in a relatively short period of time if you approach correctly.

In this article, we are going to discuss on the right strategy to study design patterns and the roadmap to get to the goal. Let's start with the studying strategy.

1. Study Strategy

1.1. Stage 1: Equip with Pattern Idioms

To be a fluent language user -- for both natural languages and artificial languages, you should be familiar with vocabularies and idioms of the language. It can be said the software patterns are idioms consisted of vocabularies which are elements of the programming language, for example, conditional statements or classes.

There are use cases frequently designed and implemented, these are solidified as patterns. If you look at the concept of the design patterns, it is a solution for a frequently encountered problem. By knowing the frequently used idioms, you are getting yourself to be prepared to be ready for future problems.

Therefore, in this first stage, it is important to absorb or get familiarized with the design patterns used in the current industry. This article provides the entries of these patterns in section two.

1.2. Stage 2: Analyze Practical Use Cases

The pattern idioms are mostly theoretical. When it comes to real-world projects, the patterns will be modified or combined for the specific requirements. It would be a good practice to analyze patterns of real-world projects to see how the patterns are actually applied.

There are popular frameworks that can be analyzed such as Java Spring, Typescript Angular, or Python Django. Or it could be open source projects like Krita, Redis, or Docker. It is a good exercise to draw the pattern diagrams for the entire project.

1.3. Stage 3: Create New Patterns

Finally, it is time to apply all the knowledge to your project. You can use, modify, or combine the existing pattern idioms; however, you can create a new one if you see patterns that don't exist in the existing idioms.

This is the goal of learning patterns since patterns make the code efficient, testable, and scalable. More you can have a chance to interpret the requirements to patterns, better you will be good at. These stages are an iterative process, which can develop your skill incrementally.

0. Conclusion

There is an infinite number of patterns in the world. The patterns should be made on-the-fly based on the specific situations encountered, rather than being made for the later use.

The main reason to study patterns in the way in which the article describes is to train your brain to see the patterns and create the programmable vocabularies and idioms for the real world. Also, this training should stop until you retire as a software engineer -- probably at the end of life.
